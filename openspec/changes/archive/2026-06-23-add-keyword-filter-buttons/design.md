## Context

前端目前爬取完成後，全部標案一次顯示在表格中，使用者無法快速聚焦在特定業務類型。本次新增四個關鍵字篩選按鈕（設備、機房、醫院、儀器），讓使用者點選後只看到與接案業務最相關的標案。

資料已在爬取後完整存於前端 state，篩選不需要重打 API。

## Goals / Non-Goals

**Goals:**
- 在表格上方顯示四個可點選的關鍵字篩選按鈕
- 點擊按鈕後，表格僅顯示標案名稱包含該關鍵字的列
- 支援多鍵同時啟用（OR 邏輯：只要命中任一關鍵字即顯示）
- 再次點擊已啟用的按鈕可取消該篩選；所有按鈕皆未啟用時顯示全部資料

**Non-Goals:**
- 自訂關鍵字輸入（固定四個，不提供 free-text 搜尋）
- 後端篩選或分頁
- 跨欄位篩選（僅針對「標案名稱」欄位）

## Decisions

### 1. 純前端篩選 vs. 呼叫後端 API
選擇**純前端篩選**。

爬取結果已完整載入 React state，在前端做陣列 filter 成本極低，不需要額外的網路往返。後端只負責爬取，保持職責單純。

### 2. OR 邏輯 vs. AND 邏輯
選擇 **OR 邏輯**（命中任一關鍵字即顯示）。

使用者目的是找出「可以接的案子」，類別之間並不互斥（例如「醫院設備」同時命中兩個關鍵字），OR 邏輯能最大化召回率，避免漏看相關標案。

### 3. 篩選狀態存放位置
使用 **React `useState`**，以 `Set<string>` 儲存目前啟用的關鍵字集合。

此狀態只影響表格顯示，不需跨元件共享，local state 即可，不需引入 Redux 或 Context。

### 4. CSV 下載範圍
**下載篩選後的資料**（目前表格顯示的列）。

使用者篩選後下載，預期是要取得當前可見的結果，若仍下載全部資料會造成困惑。

### 5. 元件結構
篩選按鈕放在現有 `App.jsx` 的爬取按鈕與表格之間，不另立新元件（規模小，額外抽元件反而增加複雜度）。

## Risks / Trade-offs

| 風險 | 緩解措施 |
|------|---------|
| 關鍵字比對大小寫不一致 | 比對時統一對標案名稱使用 `includes(keyword)`（中文不受大小寫影響，無需 toLowerCase） |
| CSV 下載篩選後資料與後端快取資料不同步 | CSV 直接從前端 filtered 陣列產生，不依賴後端 `/api/download-csv`；需評估是否改為前端直接產生 CSV 或繼續呼叫後端（見 Open Questions） |
| 篩選後資料為空 | 沿用現有「無符合條件的標案」提示即可覆蓋此情境 |

## Open Questions

- **CSV 下載**：目前後端有 `GET /api/download-csv` 端點。若改為下載篩選後資料，需決定：(a) 改由前端直接用 Blob 產生 CSV，或 (b) 將篩選後的資料 POST 給後端再下載。建議選 (a)，保持後端無狀態。
